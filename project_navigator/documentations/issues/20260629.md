**Fix round 1:** commit `fb36569` (93 tests)

## Round 2 — `apiPut is not defined`, stage UI unreachable after first goal save

### Symptoms

After round 1 was deployed, two new symptoms appeared:

1. **`apiPut is not defined` in browser console.** Clicking `+ New project`
   in the sidebar logged the project (visible in the sidebar) but threw a
   `ReferenceError`. The new project was never loaded into the main pane.
2. **"After input ultimate goal, the stage creation is not showing up."**
   Setting the goal worked locally (UI updated) but the network save
   failed silently, and the user couldn't see where to add stages.

### Root cause

`projects.js` and `stages.js` both call `apiPut` (for the active-project
flip and the goal save respectively), but neither imported it from `api.js`:

```js
// Before
import { apiGet, apiPost, apiPatch, apiDelete } from './api.js';
```

In round 1 I had verified these helpers were *exported* by `api.js` (they
were), but didn't check that they were *imported* by their callers. The
JavaScript ReferenceError was synchronous — `.catch()` on the promise
chain didn't catch it (you can't `.catch()` a ReferenceError thrown before
the promise exists).

### Downstream effects

- `createProject(name)` calls `apiPost` (worked), then `loadAndRenderSidebar()`
  (worked), then `selectProject(id)` which calls `apiPut` → throws → aborts.
  The project was created server-side but never selected client-side; the
  user saw the row in the sidebar but the main pane was stale.
- `saveGoal()` mutated `S.goal` and called `renderGoal()` *before* calling
  `apiPut`. The UI updated correctly, but the API save failed. The user
  thought the save worked; on reload, the goal reverted.

### Fix

Two lines:

```js
// frontend/js/projects.js
import { apiGet, apiPost, apiPatch, apiPut, apiDelete } from './api.js';

// frontend/js/stages.js
import { apiGet, apiPost, apiPatch, apiPut, apiDelete, ApiError } from './api.js';
```

### Empty-state hint

The "stage creation is not showing up" report was also caught by the
existing defensive guard from round 1 (toolbar hidden when no active
project) — but the message "Select or create a project from the sidebar"
didn't tell the user *where* in the sidebar to click. Updated to a
clickable hint:

```html
<div id="noProjectMsg" class="empty-stages" hidden>
  No project selected. Use the sidebar to
  <strong data-focus-new>+ New project</strong> or pick an existing one.
</div>
```

The `<strong data-focus-new>` element opens the sidebar's new-project
form when clicked. Same DOM also gets a subtle highlight-pulse animation.

### Regression test

`tests/test_frontend_imports.py` parses the import lists of every JS
file that uses `api.js` and asserts no API helper is referenced without
being imported:

```python
@pytest.mark.parametrize("filename", ["projects.js", "stages.js", "auth.js"])
def test_file_only_references_imported_api_helpers(filename):
    # ...
    missing = used - imports
    assert not missing, f"{filename} uses api helpers that are not imported: {sorted(missing)}"
```

This catches the bug at test time without needing a browser. Also asserts
that `api.js` still exports every helper, so any future drop would be
caught here before consumers break.

### Lessons

1. **Static-import checks beat runtime errors.** The `apiPut is not
   defined` would never have shipped if we'd had a test that listed every
   API helper used per file and asserted the import. We do now.
2. **Try/catch around synchronous calls.** `.catch()` on a promise chain
   doesn't help with ReferenceErrors thrown synchronously before the
   promise is constructed. Add an explicit `try/catch` if synchronous
   failure is possible.
3. **Don't say "Add your first stage" when there's no project.** Empty
   states should match the actual user goal (create a project), not the
   previous app's wording.

## Round 3 — Stage status dropdown missing

### Symptom

The user reported: "the marking drop down for stage is not working as
blockers/quesions". The stage footer had four inline buttons (To do /
Active / Blocked / Done) — not a portal dropdown like blockers and
sub-items. Even if the inline buttons worked, the inconsistency between
the two patterns was jarring.

### Root cause

The legacy `project_navigator.html` used inline status buttons for stages
(`<button class="stbtn" data-stage-status="...">`) and a portal dropdown
for blockers / sub-items (`<button class="spill" data-portal="...">`).
The frontend refactor preserved both patterns verbatim — including the
inline buttons — even though there was no functional reason to keep them
different.

Additionally, two related bugs surfaced once the dropdown was wired up:

1. `_patchItemStatus(sid, bqid, subid, body)` always assumed either
   `subid` or `bqid` was non-null. With a stage-level call (both null),
   it built `/api/projects/<pid>/blockers/null` and 404'd.
2. `openPortalMenu` showed the "Going too deep?" option for any item
   where `item.deep` was falsy — including stages, which have no deep
   mode. Sending `deep=true` against a stage would 400.

### Fix

1. **Stage footer uses the portal pill.** Replaced the 4 inline buttons
   with `pillBtn(s, s.id, '', '')`, which renders the same shape as the
   blocker / sub-item pill. Removed the now-unused
   `data-stage-status` event handler and `setStageStatus` function.

   ```js
   // Before
   <div class="status-btns">
     ${STAGE_ST.map((st) =>
       `<button class="stbtn${s.status === st ? ' ' + STAGE_ST_CLS[st] : ''}"
         data-stage-status="${s.id}|${st}">${STAGE_ST_LBL[st]}</button>`
     ).join('')}
   </div>

   // After
   <div class="status-btns">
     ${pillBtn(s, s.id, '', '')}
   </div>
   ```

2. **`_patchItemStatus` handles the stage case.** Routes fall through:
   subid → /subitems, bqid → /blockers, neither → /stages.

   ```js
   function _patchItemStatus(sid, bqid, subid, body) {
     let url;
     if (subid)      url = `/api/projects/${pid()}/subitems/${subid}`;
     else if (bqid)  url = `/api/projects/${pid()}/blockers/${bqid}`;
     else            url = `/api/projects/${pid()}/stages/${sid}`;
     apiPatch(url, body).catch(...);
   }
   ```

3. **`openPortalMenu` skips the deep toggle for stages.** An `isStage`
   flag (derived from `!bqid`) gates the "Going too deep?" /
   "Back to normal" items. Stages get the 4 NORM statuses only.

   ```js
   const isStage = !bqid;
   const statuses = (!isStage && item.deep) ? DEEP : NORM;
   // ...
   if (!isStage) {
     html += item.deep
       ? `<button class="pmitem sep-back" data-deep="false">↩ Back to normal</button>`
       : `<button class="pmitem sep" data-deep="true">⚑ Going too deep?</button>`;
   }
   ```

4. **`applyStatus` only honors `solve` for blockers / sub-items.** Stages
   never emit `solve` (it's not in the dropdown), but the guard
   `if (bqid && val === 'solve')` makes the intent explicit.

5. **Removed dead constants.** `STAGE_ST`, `STAGE_ST_LBL`,
   `STAGE_ST_CLS` are no longer exported from `state.js` and no longer
   imported into `stages.js`. The NORM list in `stages.js` is the single
   source of truth for both stages and non-deep blockers.

### Regression test

`tests/test_frontend_imports.py::test_stage_status_uses_portal_pill_not_inline_buttons`
catches the round-3 bug if anyone reintroduces inline status buttons:

```python
def test_stage_status_uses_portal_pill_not_inline_buttons():
    stages_js = (FRONTEND_JS / "stages.js").read_text(encoding="utf-8")
    assert "data-stage-status" not in stages_js
    assert "pillBtn(s, s.id, '', '')" in stages_js
    assert "isStage" in stages_js
```

### Verification

End-to-end curl PATCH against the running server:

```
Cycle through all 4 dropdown statuses (the path the portal menu now triggers):
  set → active  → active
  set → blocked → blocked
  set → done    → done
  set → todo    → todo

Backend rejects stage-only-invalid statuses (the dropdown never offers these):
  set → park    → validation - status must be one of ['todo', 'active', 'blocked', 'done']
  set → review  → validation - status must be one of ['todo', 'active', 'blocked', 'done']
  set → nice    → validation - status must be one of ['todo', 'active', 'blocked', 'done']
  set → solve   → validation - status must be one of ['todo', 'active', 'blocked', 'done']
```

Frontend dropdown matches what the backend accepts — no chance of an
"unsupported status" 400 from the dropdown.

99 tests pass.

### Lessons

1. **One status picker, one shape.** Stage / blocker / sub-item now share
   the same portal-menu machinery. Stages only render 4 options; blockers
   and sub-items can render 4 or 8 depending on `item.deep`. The picker
   code is small and the difference is a single `isStage` guard.
2. **Don't keep a UI pattern just because the legacy had it.** The legacy
   mixed inline buttons (stages) with dropdowns (blockers). That was a
   relic, not a feature.
3. **Match the dropdown's options to the API's accepted values.** Stages
   accept 4 statuses; the dropdown offers 4 statuses; sending any of the
   4 deep-only statuses returns 400. Tests on both sides (frontend +
   backend) keep this invariant.

## Round 4 — Auto-derive stage status from items

### Symptom

The user reported: "the issue is still there. And the status of stage
should be align with status from all blocker/questions under it. If all
done, stage should mark itself done, otherwise, should mark as
active/parking depending on status of items."

The manual stage dropdown (round 3) was the wrong solution: the user
wanted stage status to *automatically* track its blockers / sub-items, not
be picked by hand.

### Rollup rule (single source of truth)

Implemented identically in `backend/models.py` and `frontend/js/stages.js`:

| Condition | Derived stage status |
|-----------|----------------------|
| No blockers / sub-items | `null` — keep user-set status (no derivation) |
| All blockers + sub-items are `done` | `done` |
| Any blocker / sub-item is in deep mode (`park`, `review`, `nice`, `solve`) | `blocked` |
| Otherwise (any non-done, non-deep) | `active` |

The deep-set is exposed as `models.DEEP_STATUSES` so the frontend can
import the same set (currently mirrored in stages.js).

### Backend changes

`models.py`:

- `derive_stage_status(conn, stage_id)` → `Optional[str]`. Collects
  blockers + sub-items via a single SQL `UNION ALL` and applies the rule.
- `reconcile_stage_status(conn, stage_id)` → updates the stage row to the
  derived value (no-op if `None`, no-op if already equal). **Commits** so
  the change survives the request teardown — without this, the UPDATE
  rolled back when the connection closed.
- `update_stage(...)` now rejects `status=...` if any items exist, with
  `400 validation` "stage status is auto-derived from its blockers/sub-items".
- `create_blocker` / `update_blocker` / `delete_blocker` /
  `create_sub_item` / `update_sub_item` / `delete_sub_item` all call
  `reconcile_stage_status` after the mutation. Each captures the
  `stage_id` *before* deleting, so the reconciliation target survives the
  cascade.

### Frontend changes

`stages.js`:

- `deriveStageStatus(stage)` and `reconcileStageStatus(stage)` mirror the
  backend.
- `applyStatus`, `applyDeep`, `addBQ`, `delBQ`, `addSub`, `delSub` all
  call `reconcileStageStatus` after the local mutation so the header
  badge updates optimistically.
- Stage footer no longer renders the portal pill. It shows a non-clickable
  badge `<span class="spill ${stageBadgeCls}" style="pointer-events:none"
  title="Auto-derived from blockers and sub-items">${stageBadgeLbl}</span>`
  matching the legacy header badge style.
- The status pill click handler for stages is gone — there is no
  `pillBtn(s, s.id, '', '')` call anymore.

### Tests

Backend (`test_stages.py`, 8 new tests):

- `test_stage_status_auto_derives_to_done_when_all_items_done`
- `test_stage_status_auto_derives_to_blocked_when_any_item_deep`
- `test_stage_status_auto_derives_to_active_for_mixed_normal`
- `test_stage_status_recompute_on_sub_item_change`
- `test_stage_status_recompute_on_blocker_delete`
- `test_stage_patch_status_rejected_when_items_exist`
- `test_stage_patch_name_allowed_when_items_exist`
- `test_stage_patch_status_allowed_when_no_items`

Frontend (`test_frontend_imports.py`):

- `test_stage_status_is_auto_derived` — asserts:
  - `deriveStageStatus` and `reconcileStageStatus` are defined
  - `data-stage-status` and `pillBtn(s, s.id, '', '')` are gone
  - `reconcileStageStatus` is referenced in the mutation functions

107 tests pass (was 99).

### End-to-end smoke (curl)

```
Initial (1 blocker 'todo' + 1 sub 'todo')       → stage = active  ✓
Sub-item done (b1 still 'todo')                → stage = active  ✓
Both done                                       → stage = done    ✓
Toggle sub-item deep                           → stage = blocked  ✓
Manual PATCH stage status to 'done'            → 400 "auto-derived"  ✓
Delete the sub-item (b1 'done' alone)          → stage = done    ✓
Delete the blocker (no items)                  → stage = done (unchanged, no derivation)
```

### Lessons

1. **Auto-derived state has a single source of truth.** Backend computes;
   frontend mirrors for optimistic UX; both apply the same rule. Any
   drift shows up as a test failure or a UI flash.
2. **Recompute on every relevant mutation.** The reconciliation runs after
   *every* blocker / sub-item create / update / delete. If we'd only
   recomputed on snapshot load, the badge would lag.
3. **Make derivation no-op when the input set is empty.** Returning
   `None` for "no items" preserves the user-set status. Without this,
   deleting the last item would clobber the user's manual choice.
4. **Commit after every status-changing UPDATE.** When the connection
   closes, uncommitted writes roll back. Easy to miss because tests pass
   within a single connection.

## 1. `POST /api/projects` returns 403 — CSRF token missing or invalid

### Symptom

After registering a new account, the user clicks `+ New project` in the
sidebar. The browser console shows:

```
POST /api/projects → 403 {"code":"forbidden","error":"CSRF token missing or invalid"}
```

Same 403 for any subsequent mutation. Login with a pre-existing user fails
the same way.

### Server log (from `requirements/20260629_issue`)

```
[28/Jun/2026 21:11:14] "POST /api/auth/register HTTP/1.1" 200 -
[28/Jun/2026 21:11:14] "GET / HTTP/1.1" 200 -
[28/Jun/2026 21:11:14] "GET /api/auth/me HTTP/1.1" 200 -
[28/Jun/2026 21:11:14] "GET /api/auth/me HTTP/1.1" 200 -        ← rotates CSRF
[28/Jun/2026 21:11:27] "POST /api/projects HTTP/1.1" 403 -      ← stale token
```

### Root cause

`/api/auth/me` calls `issue_csrf()` on every request — it rotates the
session-stored CSRF token each time. The frontend has two consumers of
`/api/auth/me`:

1. `auth.bootstrap()` — called once on page load, stores the returned
   `csrf_token` in `state.csrf`.
2. `projects.loadAndRenderSidebar()` — called on every app load to refresh
   the sidebar, calls `apiGet('/api/auth/me')` and uses the response but
   **did not** update `state.csrf`.

After `loadAndRenderSidebar()` returned, the session's CSRF had been rotated
to value B, but the frontend's `state.csrf` still held value A from
bootstrap. The next `POST /api/projects` sent header `X-CSRF-Token: A`,
the server checked its session, found value B, and returned 403.

### Fix

`api.js` now extracts `csrf_token` from every successful response and calls
`setCsrf()`:

```js
// frontend/js/api.js (after response parsing)
if (payload && typeof payload.csrf_token === 'string') {
  setCsrf(payload.csrf_token);
}
```

This works regardless of which endpoint rotates the token. Today only
`/api/auth/me` rotates; if a future endpoint also rotates, the fix carries
forward without changes.

### Regression test

`tests/test_auth.py::test_csrf_rotation_does_not_break_subsequent_mutations`
reproduces the exact flow:

```python
r = client.post("/api/auth/register", json={"username": "rotuser", "password": "validpw1"})
initial_csrf = r.get_json()["csrf_token"]

# /me rotates the token in the session.
me_resp = client.get("/api/auth/me")
rotated_csrf = me_resp.get_json()["csrf_token"]
assert rotated_csrf != initial_csrf

# A mutation with the rotated token succeeds.
resp = client.post("/api/projects", json={"id": "p1", "name": "X"},
                   headers={"X-CSRF-Token": rotated_csrf})
assert resp.status_code == 200

# A mutation with the original (stale) token is rejected.
resp_stale = client.post("/api/projects", json={"id": "p2", "name": "Y"},
                         headers={"X-CSRF-Token": initial_csrf})
assert resp_stale.status_code == 403
```

## 3. Status pill dropdown does not appear

### Symptom

Clicking a status pill (e.g. `○ To do` on a blocker) does nothing — no
dropdown appears.

### Root cause

The legacy `project_navigator.html` used inline `onclick` per button:

```html
<button class="spill" onclick="openPortalMenu(event, 'sid', '', '')">To do ▾</button>
```

so `e.currentTarget` inside `openPortalMenu` was the pill button itself.

The refactor moved to event delegation on `#stageList`:

```js
el.onclick = (ev) => {
  const t = ev.target.closest('[data-portal]');
  if (t) openPortalMenu(ev, ...);  // ev.currentTarget is now the stage list, not the pill
};
```

`e.currentTarget` was the **stage list** (a large element near the top of
the page). The menu was positioned at `listBottom + 4 + scrollY` — pushed
well below the viewport, off-screen.

Two compounding bugs:

1. Wrong element used for `getBoundingClientRect`.
2. Added `scrollY` to a `position: fixed` element's top (fixed positioning
   is viewport-relative, not document-relative).

### Fix

`stages.js`:

```js
// Pass the actual pill element in.
export function openPortalMenu(btnEl, sid, bqid, subid) {
  // ...
  const rect = btnEl.getBoundingClientRect();
  pm.style.display = 'block';
  const pmW = pm.offsetWidth;
  const pmH = pm.offsetHeight;
  let left = rect.left;
  if (left + pmW > window.innerWidth - 8) left = window.innerWidth - pmW - 8;
  // Flip upward if there isn't room below the trigger.
  let top = rect.bottom + 4;
  if (top + pmH > window.innerHeight - 8) top = rect.top - pmH - 4;
  pm.style.top = top + 'px';
  pm.style.left = left + 'px';
}

// Caller (delegated handler):
const t = ev.target.closest('[data-portal]');
if (t) {
  ev.stopPropagation();
  openPortalMenu(t, sid, bqid || null, subid || null);
  return;
}
```

The upward-flip fallback also prevents the menu from being clipped at the
bottom of the viewport.

### Behavior parity with legacy

Per requirement #4 ("review the marking tag behaviors from the legacy
file, which is having the correct and expected behavior, and apply the
same logic"), the menu options match the legacy exactly:

- For a non-deep item: status options (To Do / Active / Blocked / Done) +
  separator + `⚑ Going too deep?`
- For a deep item: deep status options (Park / Review / Nice to have /
  To Solve →normal) + separator + `↩ Back to normal`

The "solve" toggle also clears `deep` (matches the legacy `applyStatus`
handler in `project_navigator.html:260-262`).

## 4. `POST /api/projects/null/stages` returns 403

### Symptom

When the user couldn't create a project (issue 1), `activeProjectId`
stayed `null`. Clicking `+ Add stage` sent `POST /api/projects/null/stages`,
which the URL rule matched and dispatched to the create-stage handler
with `pid = "null"`. The handler's `_resolve_project("null")` returned
None (the project "null" doesn't exist), so the request 404'd or 403'd.

### Fix

Defensive UI guard in `stages.js#renderStages()`:

```js
const hasProject = !!activeProjectId;
if (toolbar) toolbar.style.display = hasProject ? '' : 'none';
if (noProjectMsg) noProjectMsg.style.display = hasProject ? 'none' : 'block';
```

`index.html` gets a placeholder element that the guard reveals:

```html
<div id="noProjectMsg" class="empty-stages" hidden>
  Select or create a project from the sidebar to start adding stages.
</div>
```

This also documents the dependency between the sidebar and the stage UI
visually, instead of silently failing.

## Lessons

1. **Centralize CSRF sync.** Any endpoint that returns a rotated token must
   update the in-memory cache. Doing it in the `fetch()` wrapper (one
   place) is more robust than asking every caller to remember.
2. **`e.currentTarget` and event delegation don't mix.** Delegated handlers
   lose the original target. Pass the matched element explicitly.
3. **`position: fixed` ignores `scrollY`.** Don't add it.
4. **Hide UI affordances that depend on state.** Showing `+ Add stage` when
   `activeProjectId` is null invites a confusing 403. Render an explicit
   "no project" hint instead.
